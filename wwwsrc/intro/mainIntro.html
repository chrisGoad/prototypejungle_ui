In the simple diagram appearing in the workspace to the right, the circles derive from a common prototype, as do the arrows.

The meaning and utility of the concept of a prototype is general and fairly simple. Click here for an explanation.

X When we say that B is a prototype for A (or A is an instance of B),

we mean that whenever B has a property, such as a color, A will "inherit" that same color, unless explicitly stated otherwise. So in describing A, we don't have to repeat all the information about A that it inherits from B. Think of the effort we save in describing a scarab if we first let on that it is a kind of beetle.

A might inherit from B and B in turn from C. This is what is called a "prototype chain".

In the repository, the diagram is encoded by a prototype tree, with a node for each instance and prototype, and it has this structure:

Note that the visible circles are derived in two steps from the generic description of the circle in the repository. The middle man in this derivation is the "Local, Editable Circle Prototype", which gives control collectively over the circles in this diagram (of course, all of this goes for the arrows too). This style of derivation can capture the underlying structure of many sorts of software objects. (All JavaScript objects contain such derivations - prototype chains - internally, but once stored externally, prototype structure is gone - for example JSON discards it). The role of the prototype tree is to provide a uniform and storable encoding for the derivations, upon which repositories and applications, such as {{ProjectName}}, can be built.

The user interface gives access to both instances and prototypes. When you select (click on) a circle, the right hand panel shows two sets of properties, the properties of the instance, and the properties of its prototype (try it). Both are editable. Note that if you edit the "fill" of the prototype, all circles change color, but if you edit the "fill" of the instance only that circle changes color.
